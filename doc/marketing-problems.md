# アクセス増加施策 — Problems

## 現状の課題

プロダクト（Pythonれんしゅうちょう）はStruggling Momentに対する解決策を備えているが、**そもそもターゲットユーザーがこのプロダクトの存在を知らない**。プロダクトの価値が届かなければ「雇われる」ことはない。

---

## 1. 発見経路の不在

### P1. Struggling Momentの検索行動とプロダクトが接続していない

ターゲットユーザー（保護者・教師）がStruggling Momentに直面したとき、どのような検索行動を取るかを整理する。

| Struggling Moment | 想定される検索クエリ | 現在の接続状態 |
|---|---|---|
| SM1: 環境構築できない | 「子ども Python 簡単」「Python インストール不要」 | 未接続 |
| SM2: Chromebookで使えない | 「Chromebook Python」「ブラウザ Python」 | 未接続 |
| SM3: 英語UIで使えない | 「Python 日本語」「子ども向け プログラミング 日本語」 | 未接続 |
| SM4: エラーが読めない | 「Python エラー 日本語」「SyntaxError 意味」 | 未接続 |
| SM7: コードが消えた | （検索に至らずパニック → 離脱） | — |

**問題の本質**: プロダクトにはランディングページ、SEO対策、検索インデックスのいずれも存在しない。Struggling Momentで検索しても本プロダクトに辿り着けない。

### P2. 口コミ・共有の仕組みがない

Switch TimelineのDecidingフェーズでは、保護者同士や教師間の情報共有が重要な役割を果たす。しかし現状：

- 共有用のURLが分かりやすくない
- SNSシェアボタンがない
- 「子どもの作品を見せる」動線がない（Social Job「親に見て！と見せられる」の未活用）
- 教師が同僚に紹介するための資料がない

### P3. Passive Lookingフェーズの保護者にリーチできない

Switch Timelineの「第1の考え」段階にいる保護者は、まだ具体的な検索行動を取っていない。この層へのリーチ手段がない。

- プログラミング教育関連のコミュニティに存在感がない
- 教育系メディア・ブログでの言及がない
- SNSでの情報発信がない

---

## 2. 信頼構築の不足

### P4. Anxietyを解消する情報が不足

Four Forces分析で特定したAnxiety（不安）に対して、プロダクト上で回答する手段がない。

| Anxiety | 必要な情報 | 現在の提供状況 |
|---|---|---|
| 「本当にPythonが動くの？」 | 技術説明（Pyodide/WASM） | なし |
| 「将来移行できる？」 | 標準Python構文である旨の説明 | なし |
| 「データが消えない？」 | 自動保存の説明 | なし |
| 「安全なサイト？」 | サーバー不要・データ送信なしの説明 | なし |

保護者は子どもに使わせる前に「このサイトは安全か？」を確認したい。その判断材料がプロダクト上にない。

### P5. 教師にとっての導入根拠がない

教師がクラスで採用するには「なぜこのツールか」を説明できる必要がある。

- 学習指導要領との対応が示されていない
- 他校の導入事例がない
- 管理者（校長・教育委員会）向けの情報がない

---

## 3. Habitを崩す接点がない

### P6. Scratchユーザーへの橋渡しがない

Habit分析で最大の慣性は「Scratchで十分では？」だった。Scratchからの自然な移行パスが存在しない。

- Scratchユーザーに「次のステップ」として認知されていない
- Scratchで作ったものをPythonで書き直す導線がない
- 「Scratchとの違い」を分かりやすく示す資料がない

### P7. 既存教材との併用価値が見えない

保護者がすでに購入した書籍やオンライン教材との関係性が不明。

- 「この本の演習をここで試せます」のような連携がない
- 教科書準拠のサンプルコードがない

---

## 4. 継続利用の仕組みがない

### P8. 初回体験後のリテンションがない

Switch Timelineの「最初の体験」で成功しても、再訪問する理由がない。

- 段階的なカリキュラムやチャレンジがない
- 「次に何をすればいいか」のガイダンスがない
- 保護者向けの進捗通知がない
- ブックマーク促進の仕組みがない

### P9. コミュニティ・フィードバックループがない

ユーザーの声を集め、改善に活かし、それ自体がコンテンツになるサイクルが存在しない。

---

## 優先度の整理

プロダクトの現フェーズ（MVP完成直後）を考慮し、Effort-Impact観点で優先度を整理する。

| 優先度 | 問題 | 理由 |
|---|---|---|
| **P0** | P1: 検索経路の不在 | 最もインパクトが大きい。見つけてもらえなければ何も始まらない |
| **P0** | P4: Anxiety解消情報の不足 | 見つけても信頼できなければ使ってもらえない |
| **P1** | P2: 口コミ・共有の仕組み | 低コストで実装可能、Social Jobを活用できる |
| **P1** | P6: Scratchからの橋渡し | 最大のHabitを崩す直接的なアプローチ |
| **P2** | P8: リテンション | 初回ユーザーを獲得してから取り組む |
| **P2** | P3: Passive Lookingへのリーチ | コンテンツマーケティングは時間がかかる |
| **P3** | P5: 教師向け導入根拠 | 導入事例が必要で時間がかかる |
| **P3** | P7: 既存教材との併用 | 個別対応が必要で労力が大きい |
| **P3** | P9: コミュニティ構築 | ユーザー基盤ができてから |
