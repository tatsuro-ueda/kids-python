# 課題分析 — アクセスログ記録

## 現状の問題

### 1. ユーザー行動が見えない

現在のserver.jsは純粋な静的ファイルサーバーで、アクセスログを一切記録していない。

**困っていること:**
- 49言語のLPのうち、どの言語が実際にアクセスされているか不明
- LP → アプリ（`/app/`）への導線がどの程度機能しているか数値で把握できない
- ユーザーがどの時間帯に利用しているか分からない
- User-Agentからデバイス（PC/モバイル/タブレット）の比率が分からない

**影響:**
- 施策の優先順位を「勘」で決めるしかない
- 言語別の投資対効果が測定できない

### 2. SEO施策の効果がサーバーサイドで検証できない

GSC（Google Search Console）は検索経由のデータのみ。直接アクセスやSNS経由のトラフィックは見えない。

**困っていること:**
- リファラー情報がないため、流入元（検索/SNS/直接/Discord）が分からない
- LP改善後のアクセス数変化をリアルタイムに確認できない
- seo-tracking.mdの週次計測がGSCデータのみに依存している

**影響:**
- Discord導線やSNSシェア機能の効果が測定不能
- SEO以外のグロース施策の評価ができない

### 3. 運用上の異常を検知できない

**困っていること:**
- 404エラー（リンク切れ、クローラーの探索）がサイレントに発生している
- 異常なアクセスパターン（Bot、スクレイピング）が検知できない
- デプロイ後の動作確認が目視ベースのみ

**影響:**
- SEOに悪影響を与えるリンク切れを放置してしまう可能性
- サーバー負荷の予兆を見逃す

### 4. アクセス元の地域が分からない

49言語対応しているが、実際にどの国・地域からアクセスがあるかIPベースで把握できない。

**困っていること:**
- 言語設定と実際の地域の対応が分からない（例: `/en/` は米国？インド？）
- 地域別のアクセス傾向から次に注力すべき市場を判断できない

**影響:**
- 多言語展開の投資判断がデータに基づかない
